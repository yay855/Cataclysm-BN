603/**
604 * A vehicle as a whole with all its components.
605 *
606 * This object can occupy multiple tiles, the objects actually visible
607 * on the map are of type `vehicle_part`.
608 *
609 * Facts you need to know about implementation:
610 * - Vehicles belong to map. There's `std::vector<vehicle>`
611 * for each submap in grid. When requesting a reference
612 * to vehicle, keep in mind it can be invalidated
613 * by functions such as `map::displace_vehicle()`.
614 * - To check if there's any vehicle at given map tile,
615 * call `map::veh_at()`, and check vehicle type (`veh_null`
616 * means there's no vehicle there).
617 * - Vehicle consists of parts (represented by vector). Parts have some
618 * constant info: see veh_type.h, `vpart_info` structure and
619 * vpart_list array -- that is accessible through `part_info()` method.
620 * The second part is variable info, see `vehicle_part` structure.
621 * - Parts are mounted at some point relative to vehicle position (or starting part)
622 * (`0, 0` in mount coordinates). There can be more than one part at
623 * given mount coordinates, and they are mounted in different slots.
624 * Check tileray.h file to see a picture of coordinate axes.
625 * - Vehicle can be rotated to arbitrary degree. This means that
626 * mount coordinates are rotated to match vehicle's face direction before
627 * their actual positions are known. For optimization purposes
628 * mount coordinates are precalculated for current vehicle face direction
629 * and stored in `precalc[0]`. `precalc[1]` stores mount coordinates for
630 * next move (vehicle can move and turn). Method `map::displace_vehicle()`
631 * assigns `precalc[1]` to `precalc[0]`. At any time (except
632 * `map::vehmove()` innermost cycle) you can get actual part coordinates
633 * relative to vehicle's position by reading `precalc[0]`.
634 * Vehicles rotate around a (possibly changing) pivot point, and
635 * the precalculated coordinates always put the pivot point at (0,0).
636 * - Vehicle keeps track of 3 directions:
637 * Direction | Meaning
638 * --------- | -------
639 * face | where it's facing currently
640 * move | where it's moving, it's different from face if it's skidding
641 * turn_dir | where it will turn at next move, if it won't stop due to collision
642 * - Some methods take `part` or `p` parameter. This is the index of a part in
643 * the parts list.
644 * - Driver doesn't know what vehicle he drives.
645 * There's only player::in_vehicle flag which
646 * indicates that he is inside vehicle. To figure
647 * out what, you need to ask a map if there's a vehicle
648 * at driver/passenger position.
649 * - To keep info consistent, always use
650 * `map::board_vehicle()` and `map::unboard_vehicle()` for
651 * boarding/unboarding player.
652 * - To add new predesigned vehicle, add an entry to data/raw/vehicles.json
653 * similar to the existing ones. Keep in mind, that positive x coordinate points
654 * forwards, negative x is back, positive y is to the right, and
655 * negative y to the left:
656 *
657 * orthogonal dir left (-Y)
658 * ^
659 * -X -------> +X (forward)
660 * v
661 * orthogonal dir right (+Y)
662 *
663 * When adding parts, function checks possibility to install part at given
664 * coordinates. If it shows debug messages that it can't add parts, when you start
665 * the game, you did something wrong.
666 * There are a few rules:
667 * 1. Every mount point (tile) must begin with a part in the 'structure'
668 * location, usually a frame.
669 * 2. No part can stack with itself.
670 * 3. No part can stack with another part in the same location, unless that
671 * part is so small as to have no particular location (such as headlights).
672 * If you can't understand why installation fails, try to assemble your
673 * vehicle in game first.
674 */
